# ADR 0001: 予約管理画面 P2 再構成（IA・技術構造・ビジュアル方針の確定）

- ステータス: **Accepted**（本人承認済み 2026-07-04）
- 日付: 2026-07-04
- 対象: `src/pages/reserve/admin.astro` を中心とする管理画面一式
- 前提フェーズ: P0（挙動不変リファクタ・`docs/log/2026-07-04-p0-refactor.md`）／P1（セキュリティ基盤・`docs/log/2026-07-04-p1-security.md`）
- 実装単位: PR-1〜PR-8（本 ADR は設計確定の記録。各 PR で段階実装）

---

## Context（なぜ再構成するのか）

現状の管理画面は **単一ページ `admin.astro`**（約840行）に、認証ゲート・6機能（承認待ち／一斉送信／臨時営業休業／受付枠設定／LINE無料枠／オーナー通知）を横スクロールのメニューで詰め込んでいる。P1 で「顧客管理（全顧客の PII）」と「代理予約」を安全に載せるレール（Worker セッション層）を dev に確立したことで、次は **機能を増やしても破綻しない情報設計（IA）と、サロンの格に見合うビジュアル** へ作り替える段階に来た。

論点は3層ある: **(1) IA（何をどこに置くか）／(2) 技術構造（どう配信・ゲートするか）／(3) ビジュアル（どう見せるか）**。以下、各層で検討した選択肢・論点・理由と確定事項を残す。

---

## 決定 1: 情報アーキテクチャ（IA）

### 選択肢
- **A. 現状維持（単一ページ＋横スクロールメニュー）**: 変更コスト最小。ただし機能追加のたびに横並びが伸び、モバイルで全機能に到達しづらい。顧客管理・代理予約を足すと破綻する。
- **B. 管理トップ（ハブ）＋区分ページに分割**: トップを入口にし、業務の意味単位でページを分ける。URL が増える／ページ間遷移が発生するが、各画面が単機能で見通しが良く、拡張に強い。

### 決定: **B（ハブ＋4区分）**

管理トップ（ハブ）を入口とし、業務を4区分に整理する。**URL は全て `/reserve/admin/` 配下に置く**（後述の技術的必須条件）。

| ページ | URL | 内容 |
|--------|-----|------|
| 管理トップ（ハブ） | `/reserve/admin` | 各区分への入口＋要対応の俯瞰 |
| 予約管理 | `/reserve/admin/reservations` | 承認待ち＋**確定予約一覧**＋**代理登録** |
| 顧客管理 | `/reserve/admin/customers` | 顧客一覧・詳細（PII 段階開示） |
| 一斉通知 | `/reserve/admin/broadcast` | 一斉送信＋LINE 無料枠 |
| 設定 | `/reserve/admin/settings` | 受付枠・臨時営業/休業・オーナー通知・施術メニュー入口 |

### 理由
- 現状の6機能を「日々の予約さばき（reservations）／顧客（customers）／告知（broadcast）／設定（settings）」の**業務の粒度**で束ね直すと、オーナーの操作動線に沿う。
- **確定予約一覧**（新規要件）と**代理登録**（P0/P1 で土台を用意した `createBookingCore_` の `confirm`/`bypassWindow` 系フラグの出口）は「予約管理」に同居させ、1画面で「見る→代理で入れる」が完結する。
- `broadcast` に「LINE 無料枠」を同居させるのは、一斉送信が無料枠を消費するため「送る前に残枠を見る」動線が自然だから。

### 制約（URL を `/reserve/admin/` 配下に固定する理由）【重要】
P1 のセッション Cookie は **`Path=/reserve/admin`** で発行している（`src/worker/session.js`）。Cookie は Path 前方一致でしか送出されないため、管理ページ・管理 API は**必ず `/reserve/admin/` 配下**に置く必要がある。`/reserve/admin/customers` のようにネストする（`/reserve/customers` にはしない）。これは設計上の**動かせない制約**。

## 決定 2: 技術構造（配信とゲート）

> ※**本決定は [ADR-0002](0002-cloudflare-unify.md) により Superseded（2026-07-06・Accepted）**。「二態（dev=Worker／prod=static）＋prod はクライアントゲート」は、dev/prod とも Cloudflare 配信へ統一し**全管理ページをサーバゲートに一本化**する方針へ置き換わった。以下の記述は設計経緯の履歴として残す。**決定1（IA）・決定3（ビジュアル）は引き続き有効**。

### 選択肢（配信）
- **A. 全ページ静的（prerender）＋クライアント JS でゲート**: 現状の admin.astro 方式。prod（GitHub Pages）でも動く。ただし「HTML 自体は誰でも取得でき、JS で隠すだけ」で、PII を載せる顧客管理には弱い。
- **B. 管理ページを Worker 配信（オンデマンド）にし、サーバでゲート**: dev（Cloudflare Workers）のみ可能。P1 のセッション Cookie をサーバ側で検証してから中身を返せる。prod（静的）は Worker が無い。

### 決定: **二態（dev=Worker配信／prod=static）＋ `src/middleware.js` でゲート一元化**

- 各管理ページは **dev では `prerender=false`（Worker オンデマンド）／prod では `static`** の**二態**で配信する。
- セッション検証は各ページに散らさず **`src/middleware.js` に一元化**する（`/reserve/admin/**` へのアクセスをミドルウェアで受け、未認証はハブのログインへ）。
- 確定予約一覧は GAS 側に **`adminListConfirmed_` を新設**して供給する。
- セッション TTL は **複合化（アイドル30分＋絶対12時間）**。P1 の仮 60分（絶対のみ）から、オーナーの実運用（スマホで断続的に開く）に合わせる。

### 未解決の技術論点（PR-1 でスパイク検証する）
`prerender` を **環境で切り替える**必要があるが、Astro 6.4.8 が `export const prerender = <式>`（リテラル `true`/`false` ではなく式）を**ビルド時に静的評価するか**が未確認。評価されるなら「middleware＋prerender 二態」で素直に組める。評価されないなら**代替として dev 限定 `injectRoute`**（P1 で管理 API に既に採用済みの方式）でページも注入する。PR-1 の受け入れ条件にこのスパイク結論を含める（実装は PR-2）。

### 理由
- prod を壊さないことが最優先（P1 の教訓: アダプタ常時有効だと prod の flat `dist/` が壊れる）。二態は P1 の `isWorkerBuild` 判定を踏襲した延長線で、スタック変更ではない。
- ゲートをミドルウェアに集約すると、ページが増えても認証コードが重複せず、抜け（ゲートし忘れ）が構造的に起きにくい。

## 決定 3: ビジュアル方針（design 確定）

### 選択肢
- **A. 現状のトーンを踏襲（淡ピンク地＋モーヴ）**: LP と揃うが、管理画面としては「作業台」の緊張感・可読性に欠け、PII を扱う画面の重みに合わない。
- **B. 「カルテ棚」メタファ＋和を1段上げた管理画面**: LP（お客様向け）とは意図的にトーンを分け、オーナーの作業台にふさわしい落ち着き・情報密度・可読性を持たせる。

### 決定: **B（カルテ棚メタファ／和を1段上げた管理画面）**

design エージェントの確定案を採用（実装は後続 PR。詳細モックは別管理のデザイン作業領域に保管）。

- **色**: 漆紫ヘッダ（`#3B2A4B`→`#2C1E38`）＋金線／上質紙地（`#F2EDF0`・**ピンク地は退場**）／ゴールドは**金線・金環・印**に限定（多用しない）。
- **区分4色**: 藍鼠（reservations）／紫（customers）／古金（broadcast）／苔鼠（settings）。**朱=要対応・金=構造**の意味づけ。
- **書体**: 二書体併用（**明朝=読み物／サンセリフ tabular=数値**）。数値は等幅数字で桁を揃える。
- **PII 段階開示**: 一覧では**マスク表示**／詳細で `[表示]`→開示中は `[コピー]`＋**自動再マスク**。
- **代理予約**: 1画面で完結。

### 理由
- 管理画面は「お客様に見せる顔（LP）」ではなく「オーナーの作業台」。トーンを1段上げて情報の可読性・信頼感を優先する。
- PII 段階開示は「一覧に生の連絡先を晒さない」P1 のセキュリティ思想を **UI レベルでも徹底**するもの。表示は必要な瞬間だけ・放置で自動再マスク。

## 影響・リスク

- **prod 配信への影響なし**（PR-1 はフロント抽出のみ・二態配信は PR-2 以降。prod は従来どおり静的）。
- **GAS への影響なし**（PR-1 は GAS を触らない。`adminListConfirmed_` 新設は後続 PR）。
- **セッション Cookie の Path 制約**が IA を縛る（決定1の制約）。この前提が崩れる（Path 変更）と URL 設計を見直す必要がある。
- `prerender` 式スパイクの結果次第で PR-2 の実装方式が分岐する（middleware+二態 か injectRoute 併用か）。
