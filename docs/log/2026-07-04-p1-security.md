# 2026-07-04 予約システム大型改修 P1（セキュリティ基盤）

## 目的（なぜ）
顧客管理（全顧客のPII）と代理予約を安全に載せるための **「Worker 認可レール」** を dev に確立する。
核心の狙いは **ブラウザに『全顧客を引く長寿命トークン』を持たせない**こと。現状は管理トークン1本が
localStorage に載り、その1本で全PIIが引ける構造。これを Cloudflare Worker のサーバ認可層で置き換える。
**prod 配信（GitHub Pages）はスコープ外**。dev（既存 `wasyo-dev` Worker）で動く構成の確立に集中する。

2段階で刻む: 段階1＝GAS 単体の多層防御（Worker 非依存）／段階2＝@astrojs/cloudflare アダプタ導入＋
Worker 認可/セッション層。P0（挙動不変リファクタ・`docs/log/2026-07-04-p0-refactor.md`）の続き。

## 段階1: GAS 単体の多層防御（gas/Code.gs 内で完結）

### 1-1. 定数時間比較（double-HMAC）— `constEq_`
- 新設 `constEq_(a, b)`: 双方を `sign_()`（HMAC-SHA256→固定長 base64url）に通し、同一長ダイジェスト同士を
  バイト単位で差分積算比較（早期 return しない）。生の秘密を `===`/`indexOf` で比べると先頭一致長で処理
  時間が変わりタイミング攻撃の足がかりになるのを防ぐ。
- 適用: `verifySig_`（capability 署名）、`requireAdminToken_`（全トークンを早期break せず走査）、
  `requireDeployToken_`。挙動（正しい値は通り誤りは弾く）は不変。
- 根拠: `sign_` の出力は攻撃者に予測不能かつ常に同一長（32byte→base64url 43文字）のため、比較の
  分岐時間から元の秘密は復元不可能。

### 1-2. LINE Webhook 署名検証（fail-closed）— `verifyLineSignature_` / `handleLineWebhook_`
- 匿名到達可能な Webhook POST で詐称 events から Script Property `LINE_LAST_SOURCE` を書き換えられない
  よう、`X-Line-Signature`（チャネルシークレットで本文を HMAC-SHA256→base64）を検証。
- `verifyLineSignature_(rawBody, signature, secret)`: 同手順で再計算し `constEq_` で定数時間照合。
  LINE の署名は**標準 base64**（webSafe ではない）点に注意。
- `handleLineWebhook_`: `LINE_CHANNEL_SECRET` 未設定・署名不一致・署名取得不可はいずれも**無処理で返す
  （fail-closed）**。`doPost` から raw body を渡すよう変更。
- **【重大な制約・要把握】GAS Web App の `doPost(e)` は HTTP リクエストヘッダを参照できない**（`e` に
  headers が無い）。純 GAS 単体では `X-Line-Signature` を取得できず、本関数は**実質つねに fail-closed**に
  なる（= LINE_LAST_SOURCE の自動捕捉は事実上無効化）。これは指示（「冒頭で X-Line-Signature を検証」）
  と GAS の実体の**矛盾**。安全側（未検証は書かない）に倒しつつ、実運用の Webhook 捕捉は**段階2の
  Worker を前段に置きヘッダ検証してから GAS_ADMIN_SECRET 付きで転送**する構成で担保する設計とした
  （`lineSignature_` に body 経由フォールバックの土台を用意）。→ クロコに上告。

### 1-3. エラー秘匿
- `doGet`/`doPost` の catch が `String(err)` をそのまま返しており、スタック・内部識別子が漏えいし得た。
  クライアントには固定 `{ok:false,error:'server_error'}` を返し、詳細は `console.error`（実行ログ）へ。
- 正常系の `{ok:false,error:...}` は各ハンドラが return で返す（throw ではない）ため、既知エラー文字列は不変
  ＝フロントの分岐に影響なし。

### 1-4. 監査ログ基盤 `auditPush_` / `tokenFp_`（定義のみ・P4/P5 で配線）
- `auditPush_(operator, op, target, result)`: 顧客台帳ブックの「監査ログ」シートへ
  日時/操作者/操作/対象(マスク)/結果 を追記（`logPush_` の流儀を踏襲）。
- **生トークンは記録しない**: 操作者は呼び出し側が `tokenFp_()`（HMAC 指紋・非可逆・安定）や `via:line`
  等のラベルに変換して渡す。対象は `maskId_` で末尾4桁のみ。op は view|unmask|karteView|karteEdit|proxyBook。

### 1-5. `requireWorkerSecret_`（定義のみ・P4/P5 で配線）
- Worker だけが持つ共有秘密 `GAS_ADMIN_SECRET`（Script Property）を `body.workerSecret` で定数時間検証する
  ラッパ。GAS Web App はヘッダを読めないため body 経由。顧客系・代理予約の新エンドポイントを
  「Worker からの呼び出しに限る」ための土台。

## 段階2: @astrojs/cloudflare アダプタ導入＋Worker 認可/セッション層

### 2-0. 依存の解決（重要な経緯）
- 目標: Astro に Cloudflare アダプタを入れ、管理系だけサーバ化する。
- `@astrojs/cloudflare` 最新 14.x は Astro 7 必須。13.x が Astro 6 系。ただし **13.5.0 以降は
  `astro/app/fetch/default-handler`（Astro 6.3.0 で追加）を import** するため、既存 `astro@6.1.1` では
  `Missing "./app/fetch/default-handler" specifier` で build 失敗。
- 解決: **astro を 6.1.1 → 6.4.8（最新 6.x・既存 `^6.1.1` 範囲内の minor 上げ）に上げ、アダプタは
  13.7.0** を採用。メジャー据え置き＝スタック変更ではない。公開ページのビルド出力は回帰なし（後述ハッシュ照合）。

### 2-1. アダプタは dev 限定（prod 破壊回避）— astro.config.mjs
- **重大な設計判断**: `@astrojs/cloudflare` アダプタは出力を `dist/client/`（静的）＋`dist/server/`（Worker）
  に**再編**する。prod は GitHub Actions が `./dist` をそのまま GitHub Pages にアップロードするため、
  アダプタ常時有効だと**ルート `dist/index.html` が消え prod 配信が壊れる**。
- 対策: `isWorkerBuild = process.env.WORKER_BUILD==='1' || process.env.PUBLIC_ENV==='development'` の
  ときだけアダプタ有効＆管理系 API ルートを `injectRoute`。prod（`PUBLIC_ENV=production`）は
  **アダプタ無し＝従来の flat `dist/` 静的出力**を維持（回帰なし）。
- 管理系サーバルートは `src/pages` 直下ではなく `src/worker/routes/` に置き、**dev ビルド時のみ
  `injectRoute` で注入**（prod 静的ビルドが `prerender=false` でエラーにならないようにする）。

### 2-2. Worker セッション層（署名付き httpOnly Cookie・ステートレス）— `src/worker/`
- `session.js`: `SESSION_SECRET` を鍵に `{exp}` を HMAC-SHA256 署名した Cookie を発行/検証（Web Crypto）。
  Cookie は `HttpOnly; Secure; SameSite=Strict; Path=/reserve/admin`。TTL は `SESSION_TTL_SEC=3600`（**仮 60分**・
  本人確定後に変更する前提で定数化）。管理トークン照合 `tokenMatches` も定数時間（双方を HMAC に通す）。
- `guard.js`: オリジン制限（Origin/Referer 照合）／簡易 per-IP レート制限（CF-Connecting-IP・**ベストエフォート＝
  isolate 内メモリのみ**。厳密化は Cloudflare Rate Limiting/KV/DO を本番導入時に検討）／`Cache-Control: no-store`。
- ルート:
  - `POST /reserve/admin/api/login`: `{token, pin?}` を受け、Worker 保持の `ADMIN_TOKEN`（＋任意 `ADMIN_PIN`）と
    定数時間照合 → 短命セッション Cookie を発行。**PIN は差し込み式**: `ADMIN_PIN` 未設定なら無効（トークンのみ）、
    設定ありなら必須。既定は無効。
  - `POST /reserve/admin/api/logout`: Cookie 失効。
  - `POST /reserve/admin/api/action`: セッション検証 → allowlist の管理アクションのみ → **Worker が保持する
    `ADMIN_TOKEN` を添えて GAS `/exec` へ転送**（ブラウザは token を持たない）。

### 2-3. GAS 互換の保ち方（選択と理由）
- **選択: ADMIN_TOKENS 併存（GAS 側は無変更）**。Worker がセッション検証後、**保持する管理トークンを
  `adminToken` として GAS に送る**。GAS の `requireAdminToken_` は現行のまま＝既存6機能を壊さない最小変更。
- 理由: 全面 `GAS_ADMIN_SECRET` 化は GAS の全管理ハンドラの認可を差し替える大改修で回帰リスク大。P1 の核心は
  「ブラウザから長寿命トークンを外す」ことで、それは Worker が token を保持し Cookie だけをブラウザに渡す構成で
  達成できる。`GAS_ADMIN_SECRET`/`requireWorkerSecret_` は**顧客系・代理予約の新エンドポイント（P4/P5）**で採用する。

### 2-4. フロントのセッション化（環境判定・prod 非回帰）— admin-api.js / admin.astro
- **dev（`IS_DEV`）はセッション方式**: `apiPost` は Cookie 前提の `/reserve/admin/api/action` を叩き、
  ログインは `/login`、ログアウトは `/logout`。localStorage トークンは使わない。
- **prod は従来の localStorage＋GAS 直**（Worker が無いため）。→ **prod 管理画面は無回帰**。prod を Worker 化するかは
  「prod 配信方針」＝**今回スコープ外・要クロコ判断**。
- 6機能の呼び出し側（`apiPost(action,...)` ＋ `res.error==='forbidden'` → `handleForbidden`）は**無変更**。
  変更は `apiPost` 内部とゲート/ログインの配線のみ＝回帰面を最小化。

## 新設した秘密キー（**値はリポに書かない**・キー名のみ）
| キー | 置き場所 | 用途 |
|------|----------|------|
| `LINE_CHANNEL_SECRET` | GAS Script Properties | LINE Webhook 署名検証（Messaging API チャネルシークレット） |
| `GAS_ADMIN_SECRET` | GAS Script Properties ＋ Worker secret | Worker↔GAS 共有秘密（P4/P5 の新エンドポイント保護） |
| `SESSION_SECRET` | Worker secret（`.dev.vars`／`wrangler secret`） | セッション Cookie の署名鍵 |
| `ADMIN_TOKEN` | Worker secret（`.dev.vars`／`wrangler secret`） | Worker が保持する管理トークン（GAS の `ADMIN_TOKENS` の1つと一致させる） |
| `ADMIN_PIN` | Worker secret（任意・未設定で無効） | ログイン時の追加 PIN |
| `ALLOWED_ORIGIN` | Worker var（任意） | オリジン制限の許可リスト（未設定なら同一オリジンのみ） |

> 既存の `PUBLIC_RESERVE_API`（GAS /exec URL）は Worker からも参照する（`RESERVE_API` 名でも可）。

## 検証結果
### ビルド（回帰確認）
- **prod モード**（`pnpm build`・アダプタ無し）: **flat な `dist/` に7ページ生成**（従来どおり）。GitHub Pages の
  `./dist` アップロード前提を維持＝**prod 配信は無回帰**。公開静的ページのハッシュ照合（P0 baseline 比）:
  - **byte 一致（4/6）**: `reserve/manage`・`reserve/decision`・`reserve/message`・`privacy`。
  - **差分（2/6）**: `index.html`・`reserve/index.html`。両者が参照する `_astro` 資産は**抽出 CSS バンドル1つのみ**で、
    差分は**content-hash 済み CSS ファイル名の変化のみ**（`astro 6.1.1→6.4.8` の CSS 処理由来）。`.astro` ソースは未変更のため
    **レンダリング内容は不変**（＝機能回帰ではない・cosmetic）。※実ブラウザのピクセル一致は未検証。
- **dev モード**（`WORKER_BUILD=1 pnpm build`・アダプタ有り）: `dist/client`（静的）＋`dist/server/entry.mjs`（Worker）＋
  login/logout/action チャンクを生成。`.wrangler/deploy/config.json`（→`dist/server/wrangler.json`）のリダイレクトで
  `npx wrangler deploy` が使う設定を自動生成。

### GAS（段階1）
- `node --check`（`.js` コピー）**合格**（各コミット時）。
- コード対照でロジック説明可能（定数時間比較＝double-HMAC で早期 return しない／Webhook＝本文 HMAC の base64 を
  constEq_ で照合し未検証は無処理／エラー秘匿＝catch は server_error 固定＋console へ詳細）。
- **未確認**: GAS はローカル実行不可のため実行時挙動は静的対照止まり（clasp push/デプロイはしない方針）。
  **X-Line-Signature は GAS がヘッダを読めず取得不可＝Webhook 署名検証は実質つねに fail-closed**（段階2の Worker 前段で担保）。

### Worker 認可（段階2・`astro dev`＋`.dev.vars` のダミー値・RESERVE_API は死にポートで実 GAS/PII 不使用）
実測（`Origin: http://localhost:4321`）:
1. 未ログインで `POST /action` → **401 `forbidden`**（GAS を叩かずセッションゲートで拒否）。
2. 誤トークンで `POST /login` → **401 `invalid_credentials`**。
3. 正トークンで `POST /login` → **200 `{ok:true}` ＋ `Set-Cookie: wasyo_admin_session=…; Max-Age=3600; Path=/reserve/admin;
   HttpOnly; Secure; SameSite=Strict`**（payload は `{"exp":…}`＝60分 TTL）。
4. セッション付きで `POST /action` → **502 `upstream_error`**（＝**セッション検証を通過**し GAS 転送段へ到達。死にポートで失敗）。
5. セッション付き＋非 allowlist アクション → **400 `unknown_action`**。
6. 改ざん Cookie → **401 `forbidden`**（署名検証で偽造を拒否）。
7. クロスオリジン `/login` → **403 `forbidden_origin`**（オリジンガード）。加えて Astro 標準 checkOrigin も
   フォーム型 POST のクロスサイトを 403 で拒否（多層）。
- **未確認**: 実 Cloudflare（`wasyo-dev` Worker）へのデプロイと、既存6機能の実 GAS 越しの end-to-end
  （push が要る＝クロコが本人確認して実施）。実ブラウザでの管理画面 UI 操作。
