## 実装の要点と設計判断
### 1. 管理ページのオンデマンド化 = literal `export const prerender = false`（injectRoute 移動はしない）
- ADR-0002§54／MIGRATION§1 は文面上「管理ページの injectRoute」を挙げるが、これは**二態（環境で prerender を切替）が必要だった名残**。統一で**無条件 `false`** になったため、式ではなく **literal `false`** で足りる。
- **PR-1 スパイクで実証済み**: literal `false` はオンデマンド化される（`spike-false` は静的 HTML が出ない）。静的評価されず prerender=true に黙ってフォールバックするのは**非リテラル式のときだけ**。したがって `src/admin-pages/` への移動＋`injectRoute` は不要で、`src/pages/reserve/admin.astro` に留めて1行で足りる（ファイル移動・import 深さ変更のリスクを回避）。
- **なぜオンデマンドが必須か**: 静的アセットは Cloudflare が Worker を介さず ASSETS から直接配信するため、静的化すると middleware（サーバゲート）が走らない。オンデマンド化して Worker を通すことでゲートが効く。

### 2. ログイン導線＝独立した公開ページ（`/reserve/admin/login`）
- サーバゲートで「未認証者に管理ページ HTML を返さない」を満たすと、従来ページ内に埋め込んでいたログインフォームが未認証者から到達不能になる。そこで**公開のログインページを分離**した。
- ログインページは **prerender 静的（公開）**。ミドルウェアはオンデマンド管理ページのみをゲートするため、静的なログインページは ASSETS から直接配信され、ゲート対象外で自然に公開される（＝鶏卵問題が起きない）。PII は一切持たない。
- 中身は旧 `AdminGate` の #gate マークアップ・admin.css を再利用（見た目一貫）。

### 3. middleware の env 取得は動的 import（ビルド時プリレンダ対策）
- middleware は**静的ページのプリレンダ時（ビルド時）にも実行**される。Worker 専用の `cloudflare:workers` を top-level static import すると、ビルド時のロードで壊れる懸念があった。
- そこで**パス除外の後に `await import('cloudflare:workers')`** で実行時（Worker 上）にだけ env を解決する。管理ページはオンデマンド＝Worker ランタイムでのみここに到達するため安全。MIGRATION§1 は「動的 import ガードは不要化できる」としていたが、それは実行時の env 可用性の話で、**ビルド時プリレンダの安全性は別論点**のため動的 import を採った。ビルドは dev/prod 相当とも成功を確認。

### 4. リダイレクト先は末尾スラッシュ（canonical）
- 未認証リダイレクト先を `/reserve/admin/login`（スラッシュ無し）にすると、Cloudflare の ASSETS 層が `/login → /login/` の 307 を挟む（無害だが1ホップ余分）。静的アセットの canonical 形に合わせ **`/reserve/admin/login/`** へ直接リダイレクトして省いた（handleForbidden・ログアウトも同様）。

### スコープ厳守（今回やっていないこと）
push なし／本番 DNS・GitHub Pages・Cloudflare ダッシュボード操作なし／prod Worker 秘密の登録なし／GAS 不変。ADR-0001 の**決定1（IA・Path=/reserve/admin 制約）は不変**（管理系は全て `/reserve/admin/` 配下）。

## 検証（実体ベース）
### ビルド（adapter 常時有効・PR-A 済みの上）
- **prod 相当**（`PUBLIC_ENV=production pnpm build`）・**dev 相当**（`WORKER_BUILD=1 PUBLIC_ENV=development pnpm build`）とも成功。
- **出力構造**（prod 相当）:
  - `dist/client/reserve/admin/index.html` → **不在**（＝管理ページはオンデマンド・Worker/middleware を通る）。
  - `dist/client/reserve/admin/login/index.html` → **存在**（＝ログインページは静的・公開）。
  - 公開ページ（index/privacy/reserve/decision/manage/message）→ すべて `dist/client` に静的出力（**prerender 静的を維持**）。
  - `dist/server/virtual_astro_middleware.mjs` に middleware がバンドルされ、`verifySession`／`reserve/admin/login`／`SESSION_SECRET` を含むことを確認。

### 認証判定ロジックの単体検証（session.js・テスト秘密はメモリ内のみ・ファイル生成なし）
`issueSession`→`verifySession` の往復で fail-closed を確認: 有効=true／署名改ざん=false／別秘密=false／空トークン=false／**秘密未設定(env空)=false**／期限切れ=false（全ケース期待どおり）。秘密未設定でも false に落ちる＝**fail-open しない**。

### サーバゲートの HTTP E2E（`wrangler dev` で built Worker をローカル起動・秘密は `--var` の使い捨てテスト値・ファイル生成なし）
ASSETS（静的）と Worker（オンデマンド）が本番同様に分離される構成で確認:
- 未認証 `GET /reserve/admin` → **302 → Location `/reserve/admin/login/`**、**本文 0 バイト・管理 HTML（adminMenu/承認待ち）の痕跡ゼロ**（＝HTML を返さない）。追跡で 200・ログインフォーム表示。
- `GET /reserve/admin/login/` → 200（ログインフォーム）。公開ページ `GET /` → 200（ゲート対象外）。
- `POST /api/login` 誤トークン → 401／正トークン → 200 ＋ `Set-Cookie: wasyo_admin_session=…; Path=/reserve/admin; HttpOnly; Secure; SameSite=Strict`。
- 認証済み `GET /reserve/admin`（Cookie 付き）→ **200・管理 HTML 本文あり**（adminMenu/承認待ち/ログアウト・7407 bytes）。
- 改ざん Cookie → **302・本文 0**（fail-closed）／Cookie 無し → 302。
- 未認証 `POST /api/action` → **401 forbidden**（middleware は素通しし、route が自前でセッション検証）。
