## `prerender` 式スパイクの結論（PR-2 ゲート方式の可否）
**問い**: PR-2 の dev/prod 二態を `export const prerender = <式>`（環境で true/false 切替）で組めるか？
Astro 6.4.8 がページ frontmatter の**非リテラル式**をビルド時に静的評価するか。

**実験**: worker ビルド（アダプタ有り・`PUBLIC_ENV=development`）に3ページを投入:
- `spike-true`（リテラル `true`）→ 静的プリレンダ（`dist/client/spike-true/index.html` あり）
- `spike-false`（リテラル `false`）→ オンデマンド（静的 HTML なし）
- `spike-expr`（`prerender = import.meta.env.PUBLIC_ENV !== 'development'` ＝この環境では **false 評価されるべき**）
  → **静的プリレンダされた**（`dist/client/spike-expr/index.html` あり）。ビルドエラーにはならない。

**結論**: **Astro 6.4.8 は `export const prerender = <非リテラル式>` を静的評価しない**。リテラルでないと
判定できず、**黙って prerender（静的）にフォールバック**する（false にすべき式が静的化された）。
→ **「1つの admin ページ file で式により prerender を環境切替」する方式は不可**（dev でも静的化され Worker
オンデマンド化＝サーバゲートが効かない）。

**PR-2 の採るべき方式**: **dev 限定 `injectRoute`**（P1 で管理 API に既に採用済みの方式）。管理ページを
`src/pages` の自動ルーティングから外し、`astro.config` の integration で**config 時の Node ブール
（`isWorkerBuild`）を `injectRoute({ prerender })` に渡して二態化**する（config 時に解決されるため式の
静的評価問題を回避）。middleware によるセッションゲートはオンデマンド（dev）でのみ走る点も P1 の
「dev=サーバゲート / prod=クライアントゲート」と整合。※実装は PR-2（本 PR ではしない）。
