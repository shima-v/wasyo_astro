# 2026-07-04 予約システム大型改修 P0（挙動不変の共通リファクタ）

## 目的（なぜ）
予約システムの大型改修に向けた**第1フェーズ P0**。この段階では**新機能を一切足さず**、後続フェーズ

## 1. GAS: createBookingCore_ の抽出

### 導入した opts フラグと分岐点（既定＝現行再現）
| フラグ | 既定 | 分岐する箇所（抽出前の行） |
|--------|------|---------------------------|
| `requireContact` | `true` | 連絡手段の必須チェック（`missing_required`/`missing_email`/`missing_contact`） |
| `requireReferrer` | `true` | 男性 `gender==='male'` → 紹介者 `referrer` 必須（`referrer_required`） |
| `confirm` | `false` | 【仮】/【確定】・色 ORANGE/GREEN・`status` pending/confirmed・生成直後の `ledgerUpsert_` の有無 |
| `bypassWindow` | `false` | リードタイム/受付上限（`too_soon`/`too_far`）・`isOpenDay_`/`isValidStart_`/`isClosedSlot_`（`slot_closed`）のスキップ |
| `notifyOwnerPending` | `true` | `notifyOwnerNewBooking_`（仮予約のオーナー通知）送信の有無 |

**ダブルブッキング防止 `overlapsBusy_`（`slot_taken`）は opts に関係なく常に実施**（意図的に分岐外）。

この段階では**新エンドポイント（adminCreateBooking_）は作らない**。フラグ分岐の骨組みだけを入れ、
既定パスが現行と等価であることをコードで保証する。

### createBooking_ 等価性の根拠（フラグ分岐ごとの対照）
既定は `createBooking_(b)` → `createBookingCore_(b, DEFAULT_CREATE_OPTS)`。opts は
`!== false`/`=== true` で解決するため、既定オブジェクトでも `{}` でも全既定になる。各既定値での対照:

- `requireContact = true`：抽出前の連絡先3チェックを `if (requireContact) { ... }` で囲っただけ。3文の
  内容・順序・返す error（`missing_required`/`missing_email`/`missing_contact`）は文字列一致で不変。
  → 既定 true で必ず全チェックが走る＝等価。
- `requireReferrer = true`：`gender==='male' && !referrer` チェックを `if (requireReferrer)` で囲っただけ。
  抽出前と同位置（連絡先チェックの直後・`missing_slot` の前）で実行される＝等価。
- `bypassWindow = false`：`too_soon`/`too_far`（リードタイム・受付上限）と `slot_closed`（営業日/開始時刻/
  休枠）を各々 `if (!bypassWindow) { ... }` で囲っただけ。既定 false で必ず両ブロックが走る＝等価。
- `confirm = false`：`status = confirm ? CONFIRMED : PENDING` は false で `PENDING`。タイトル接頭辞は false で
  `'【仮】'`、色は false で `ORANGE`。`if (confirm) ledgerUpsert_(...)` は false で**呼ばれない**（抽出前も
  作成時は台帳更新せず、承認 `decide_` 時のみ更新）。戻り値 `status` も PENDING。→ 生成物・戻り値すべて等価。
- `notifyOwnerPending = true`：`notifyOwnerNewBooking_(...)` を `if (notifyOwnerPending)` で囲っただけ。
  既定 true で必ず送る＝等価。
- 上記フラグに関与しない部分（menu 検証・`missing_slot`・`ledgerKey_`/`ledgerLookup_`/`effectiveMenu_`/
  `slotMin_`・カレンダー取得・`LockService`・`overlapsBusy_`・`setEventProps_` に渡す props・
  `setDescription`・`notifyCustomer_`・戻り値の他フィールド）は**文字通り同一**。
- `setEventProps_` に渡す props は抽出前のオブジェクトリテラルを変数 `evProps` に名前付けしただけで、
  キーの内容・挿入順（＝`Object.keys` の列挙順→`setTag` 呼び出し順）は不変。

**未確認事項**：GAS はローカル実行不可のため、上記は**コード対照による静的等価性の担保**。dev GAS/エディタでの
実行確認は未実施（clasp push・デプロイはしない方針のため。実行確認が要るならクロコ経由で本人に相談）。

## 2. フロント: admin-api.js の抽出
- トークン保管（`getToken`/`setToken`/`clearToken`、キー `wasyoAdminToken`）はモジュールトップの named export。
- `createAdminApi({reserveApi, el})` が `apiPost`/`showGate`/`showPanel`/`handleForbidden` を返すファクトリ。
  DOM 参照は `el`（`document.getElementById` ラッパ）**越しに都度解決**するため、抽出前の遅延ルックアップ挙動を
  そのまま保つ（要素参照を先取りキャッシュしない）。
- `apiPost` は POST（`text/plain`＝CORS プリフライト回避）・`adminToken` 添付・`opts.silent` で
  `runExclusive`（送信オーバーレイ＋二重送信ガード）を素通し、まで抽出前と同一ロジック。
- **トークン方式（localStorage）は現行のまま**。Worker セッション化は後続フェーズ（P1）で行い、ここでは純粋な抽出のみ。
- admin.astro 側は import＋ファクトリ呼び出しに置換。旧 `TOKEN_KEY`/`getToken`/`setToken`/`clearToken`/
  `apiPost`/`showGate`/`showPanel`/`handleForbidden` の定義を削除。呼び出し側（7つの送信操作・ゲート配線・起動）は
  同名・同シグネチャのため無変更で動作。

## 3. フロント: availability.js の抽出
- `fetchAvailabilityRaw(reserveApi)`：`?action=availabilityRaw` を fetch → `data.ok` 検証 →
  `{days, holidays}` を返す（失敗時は throw）。抽出前の try 内ロジックと同一。
- `computeDaysForSlot(raw, slotMin)`：素材＋slotMin から予約可能日時を導出する**純粋関数**（DOM/state 非依存）。
  半開区間の重なり判定 `tStart < bEnd && bStart < tEnd` は文字通り同一。
- index.astro 側は薄いラッパに:
  - `prefetchAvailabilityRaw`：`state.raw = await fetchAvailabilityRaw(RESERVE_API)` に置換。
    `rawFetching` ガード・`onRawReady` コールバック・catch で `state.raw=null` の**オーケストレーションは index に残す**。
  - `buildDaysForSlot_`：**raw 未取得時に `state.holidays` を触らない**従来挙動を明示的に維持
    （`if (!state.raw) { state.days = []; return; }` を先に置き、`computeDaysForSlot` は raw ありのときだけ呼ぶ）。
    ※抽出前も `buildDaysForSlot_` は `state.raw` あり分岐でしか呼ばれないため、この差は実挙動には現れないが念のため保存。
- 後続フェーズ（代理予約フォーム）は `fetchAvailabilityRaw`＋`computeDaysForSlot`（データ層）を再利用できる。
  UI 描画（`renderCalendar`/`renderDates`）は index 固有 DOM に密結合のため今回は抽出せず、index 挙動不変を優先。
